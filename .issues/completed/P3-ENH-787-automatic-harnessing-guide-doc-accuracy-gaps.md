## Summary

`docs/guides/AUTOMATIC_HARNESSING_GUIDE.md` has three accuracy/completeness gaps: the wizard H3 step implies `check_mcp` is wizard-generated (it isn't), the `check_stall` YAML example references `skip_item` without defining it, and the `check_stall` placement note is inconsistent with its YAML example.

## Current Behavior

Three issues found during audit:

### 1. `check_mcp` not surfaced by the wizard (Accuracy, Medium) — lines 13, 284–299

The guide states "Each harness applies up to five evaluation phases" (line 13) and Step H3 (line 284) says the wizard "presents only relevant options. All available phases are pre-selected." This implies the wizard offers all 5 phases including `check_mcp`.

In reality, `skills/create-loop/loop-types.md` Step H3 only presents **4 options** (Tool-based gates, LLM-as-judge, Diff invariants, Skill-based evaluation) — no `check_mcp`. The `check_mcp` phase must be added manually after wizard generation. The guide never says this.

**Fix**: Add a note in Step H3 clarifying that `check_mcp` is not wizard-generated and must be inserted manually into the output YAML, with a pointer to the `check_mcp` section.

### 2. `check_stall` example missing `skip_item` state definition (Completeness, Low) — lines 459–467

The `check_stall` YAML example ends with `on_no: skip_item` but never shows the `skip_item` state that must exist in the FSM. Without it the FSM is invalid. `skills/create-loop/reference.md` (line 718) shows:

```yaml
skip_item:
  action: echo "Skipping ${current_item} after 3 failed retries"
  action_type: shell
  next: done
```

**Fix**: Append a `skip_item` state definition to the `check_stall` YAML example, or add a note pointing users to define it.

### 3. `check_stall` placement note inconsistent with YAML example (Accuracy, Low) — lines 456–467

Placement note says: *"Insert `check_stall` between `execute` and the first check state (or between `check_invariants` and `advance`)."*

The YAML example shows `on_yes: advance` — only valid for the second placement. The first placement would need `on_yes: check_concrete` (or the first check state).

**Fix**: Show both placement variants or align the placement note with the single example shown.

## Expected Behavior

- Step H3 section notes that `check_mcp` must be added manually (not generated by wizard)
- `check_stall` YAML example includes `skip_item` state definition or prose note
- `check_stall` placement note is consistent with the YAML example routing

## Impact

- **Priority**: P3 - Documentation accuracy issue; affects users building harness loops who may be confused about `check_mcp` availability or produce invalid FSMs from the `check_stall` example
- **Effort**: Small - Three targeted prose/YAML edits in a single guide file
- **Risk**: Low - Documentation-only change; no code changes
- **Breaking Change**: No

## Scope Boundaries

- Only `docs/guides/AUTOMATIC_HARNESSING_GUIDE.md` needs changes
- Do not restructure or rewrite the guide; only fix the three identified accuracy/completeness gaps
- Do not add new sections; fix in-place

**Completed**: 2026-03-16

Three targeted edits made to `docs/guides/AUTOMATIC_HARNESSING_GUIDE.md`:
