# FEAT-2078: Add distill-traces skill to extract reusable loop fragments from history

## Summary

Adds a `distill-traces` skill that mines successful loop execution traces from `.ll/history.db` into reusable YAML state templates and transition fragments, giving future loop authors a concrete scaffold library rather than relying on prose documentation or manual history review.

## Current Behavior

No skill exists to extract or reuse patterns from past successful loop runs. Developers who want to reuse a known-good state sequence must manually sift through session history or loop YAMLs, with no structured artifact library of loop primitives available.

## Expected Behavior

Running `/ll:distill-traces <loop-name>` queries history for successful runs of that loop, extracts action sequences and transition patterns, and writes reusable YAML fragments to `loops/lib/<loop-name>/` alongside a `primitives.md` index. An optional `--update-suggester` flag pushes the extracted patterns into loop-suggester's context for future suggestions.

## Motivation

Successful loop execution traces already exist in `.ll/history.db` and ll-logs, but no skill mines them into reusable loop YAML fragments or Python helpers. Distilling behavioral traces into a small reference library ŌĆö rather than documenting strategy in prose ŌĆö gives future loop runs a concrete scaffold to build on.

## Use Case

A developer wants to reuse patterns from past successful runs of `rn-plan`. They invoke `/ll:distill-traces rn-plan --min-success 3` and receive YAML state templates and transition patterns in `loops/lib/rn-plan/`, along with a `primitives.md` summary they can reference when authoring or modifying the loop.

## Proposed Solution

Create a `ll:distill-traces` skill that:
1. Traverses `.loops/.history/*-<loop_name>/` to find candidate runs (pattern from `scripts/little_loops/fsm/persistence.py:_RUN_FOLDER` regex `r"^(\d{4}-\d{2}-\d{2}T\d{6})-(.+)$"`)
2. Filters for successful runs: reads each `state.json`, keeps runs where `status == "completed"` and `terminated_by == "terminal"`; applies `--min-success` threshold
3. Extracts state sequences from `events.jsonl` by collecting `state_enter` events and `route` events (`{"event": "route", "from": ..., "to": ...}`) across qualifying runs
4. Aggregates the most frequent state sequences and action strings into YAML fragment entries following the `fragments:` schema defined in `scripts/little_loops/loops/lib/common.yaml`
5. Writes output to `scripts/little_loops/loops/lib/<loop-name>/state-templates.yaml` and `transitions.yaml`, with a `primitives.md` index
6. Defers `--update-suggester` to a follow-up issue ŌĆö no context-update mechanism exists in loop-suggester (confirmed by codebase research)

### Data Access Path

`ll-session` has no subcommand that filters by `loop_name` or success status ŌĆö `ll-session recent --kind loop` returns all events without a loop-name filter. The correct data access path is direct filesystem traversal:

```bash
# Find all runs for a loop:
ls .loops/.history/ | grep -- "-<loop-name>$"

# For each candidate run dir:
#   .loops/.history/<YYYYMMDDTHHMMSS>-<loop_name>/state.json   ŌåÆ success check
#   .loops/.history/<YYYYMMDDTHHMMSS>-<loop_name>/events.jsonl  ŌåÆ trace extraction
```

Alternatively, raw SQL against `.ll/history.db` can identify successful timestamps:
```sql
SELECT ts FROM loop_events
WHERE loop_name = ? AND transition = 'loop_complete' AND state = 'terminal'
```
But the `events.jsonl` filesystem source is richer (includes `action_complete`, `evaluate` events not stored in the DB).

### Fragment Output Schema

Output YAML must follow the `fragments:` map structure from `scripts/little_loops/loops/lib/common.yaml`:

```yaml
# Generated by /ll:distill-traces <loop-name>
# Import with: import: - lib/<loop-name>/state-templates.yaml
fragments:
  <state_name>:
    description: |
      Extracted from N successful runs of <loop-name>.
      Caller must supply: on_yes, on_no (and optionally on_error, timeout).
    action_type: shell|prompt|slash_command
    action: |
      <most common action string across qualifying runs>
    evaluate:
      type: exit_code|llm_structured|output_contains
```

```bash
# Example invocation
/ll:distill-traces rn-plan --min-success 3
# ŌåÆ writes scripts/little_loops/loops/lib/rn-plan/primitives.md
# ŌåÆ writes scripts/little_loops/loops/lib/rn-plan/state-templates.yaml
# ŌåÆ writes scripts/little_loops/loops/lib/rn-plan/transitions.yaml

/ll:distill-traces rn-plan --min-success 5
# ŌåÆ same, with higher success threshold
```

### Codebase Research Findings

_Added by `/ll:refine-issue` ŌĆö based on codebase analysis:_

- `ll-session` has **no loop-name-filtered subcommand** ŌĆö `recent --kind loop` returns all events without a loop-name or status filter; direct `.loops/.history/` traversal is the correct primary path
- `loop_events` table columns relevant to success detection: `loop_name`, `transition` (use `'loop_complete'`), `state` (holds `terminated_by` value for complete rows, mapped in `SQLiteTransport.send()`)
- The `_LOOP_EVENT_TYPES` frozenset in `session_store.py` shows which events reach the DB: only `loop_start`, `loop_resume`, `loop_complete`, `state_enter`, `route`, `retry_exhausted`, `cycle_detected`, `max_iterations_summary` ŌĆö action-level detail (`action_complete`, `evaluate`) is `events.jsonl`-only
- No `loops/lib/<loop-name>/` subdirectory pattern exists yet; the current flat domain-scoped files (`common.yaml`, `cli.yaml`, `harness.yaml`) set the schema, and the per-loop subdirectory is a new convention this feature introduces
