# FEAT-1007: Add `apply` Sub-Command to `ll-deps` for Writing Dependency Relationships

## Summary

Add an `apply` sub-command to `ll-deps` that writes `blocked_by` / `blocks` dependency fields into issue frontmatter based on the proposals generated by `ll-deps analyze`. Bridges the gap between proposal generation (read-only) and actually wiring dependencies into issue files.

## Current Behavior

`ll-deps analyze` proposes dependency relationships with confidence scores and rationale, but all proposals are read-only output. Users must manually edit each issue file's frontmatter to add `blocked_by` / `blocks` fields. `ll-deps fix` handles backlinks and stale refs but cannot add new relationships — it only repairs existing ones.

## Expected Behavior

`ll-deps apply` re-runs the analysis internally and writes accepted proposals (above a confidence threshold) directly into issue frontmatter, then reports what was written. A `--dry-run` flag previews changes without modifying files. An explicit pair form lets users manually assert a single relationship without editing files.

Proposed interface:
```
ll-deps apply                              # Apply proposals ≥ 0.7 confidence
ll-deps apply --min-confidence 0.5         # Lower threshold
ll-deps apply --dry-run                    # Preview only (no writes)
ll-deps apply --sprint my-sprint           # Sprint-scoped
ll-deps apply FEAT-001 blocks FEAT-002     # Manual explicit pair
```

After `apply`, users can run `ll-deps fix` to add any missing backlinks.

## Motivation

The `analyze` → `apply` → `fix` pipeline closes the loop on dependency management. Without `apply`, every run of `analyze` produces proposals that go unacted on — the tool identifies relationships but provides no mechanism to act on them, making the feature half-baked. The explicit pair form is particularly valuable for manual dependency assertions in cases where file-overlap heuristics would not infer the relationship.

## Proposed Solution

Add an `apply` subparser to `main_deps()` in `scripts/little_loops/cli/deps.py`:

1. **Implicit-apply mode** (no pair args): call `analyze_dependencies(...)`, filter proposals by `--min-confidence`, then write `blocked_by` fields to source issue frontmatter via the existing `frontmatter.py` utilities.
2. **Explicit-pair mode** (`FEAT-001 blocks FEAT-002`): validate both IDs exist, then directly write the relationship without running analysis.
3. Backlinks are NOT added by `apply` — user should run `ll-deps fix` afterward (consistent with existing tool separation).
4. `--dry-run` prints what would be written, with no file mutations.

Key reuse: `little_loops.frontmatter` for frontmatter read/write, `little_loops.dependency_mapper.analyze_dependencies` for proposals.

## Use Case

A developer runs `ll-deps analyze` and sees 4 proposed dependencies with confidence 0.8+. Instead of opening each issue file and manually adding `blocked_by` fields, they run `ll-deps apply --dry-run` to preview, then `ll-deps apply` to write all relationships at once. They follow up with `ll-deps fix` to add backlinks, then `ll-deps validate` to confirm a clean state.

## Acceptance Criteria

- [ ] `ll-deps apply` (no args) applies proposals ≥ 0.7 confidence and reports each write
- [ ] `--min-confidence` adjusts the threshold
- [ ] `--dry-run` prints proposed changes without modifying any files
- [ ] `--sprint` restricts to issues in the named sprint
- [ ] Explicit pair form (`ll-deps apply FEAT-001 blocks FEAT-002`) writes the relationship directly
- [ ] Invalid issue IDs in explicit pair form produce a clear error and exit non-zero
- [ ] Already-present relationships are skipped (idempotent), not duplicated
- [ ] Backlinks are intentionally NOT written (user runs `ll-deps fix` for that)
- [ ] All new code paths covered by tests

## API/Interface

```python
# New argparse subcommand in main_deps()
apply_parser = subparsers.add_parser("apply", help="Write dependency relationships to issue frontmatter")
apply_parser.add_argument("source", nargs="?", help="Source issue ID for explicit pair (e.g. FEAT-001)")
apply_parser.add_argument("relation", nargs="?", choices=["blocks", "blocked-by"], help="Relationship direction")
apply_parser.add_argument("target", nargs="?", help="Target issue ID for explicit pair (e.g. FEAT-002)")
apply_parser.add_argument("--min-confidence", type=float, default=0.7, help="Minimum confidence to apply (default: 0.7)")
apply_parser.add_argument("--dry-run", "-n", action="store_true", help="Preview without writing")
apply_parser.add_argument("--sprint", type=str, default=None, help="Restrict to sprint issues")
```
