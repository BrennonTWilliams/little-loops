# FEAT-919: JSON Schema Generation for Event Types

## Summary

Generate JSON Schema files for all 19 `LLEvent` types so non-Python tools can validate event payloads without parsing `docs/reference/EVENT-SCHEMA.md` by hand. Schemas are committed to `docs/reference/schemas/` alongside the human-readable schema doc and regenerated by a script when events change.

## Context

Split from FEAT-916 (Extension SDK). FEAT-916 resolved that the scaffolded `extension.py` skeleton links to `docs/reference/EVENT-SCHEMA.md` in a comment â€” no schema file reference in templates. This issue covers the schema generation artifact itself, targeting non-Python tooling authors as the primary consumer.

## Current Behavior

No machine-readable schema exists. Extension authors and non-Python tools must parse `docs/reference/EVENT-SCHEMA.md` manually to understand event payload structure.

## Expected Behavior

- `docs/reference/schemas/` contains one `.json` schema file per event type (19 total)
- A generation script at `scripts/little_loops/generate_schemas.py` (or a CLI flag) regenerates all schema files from the known event type catalog
- Schema files are valid JSON Schema (draft-07 or later)

## Use Case

**Who**: Non-Python tooling author (dashboard builder, log analyzer, monitoring integration)

**Context**: Needs to validate or process `LLEvent` payloads in a non-Python tool and wants a machine-readable contract rather than parsing `docs/reference/EVENT-SCHEMA.md` manually

**Goal**: Import a JSON Schema file for a specific event type (e.g., `state_enter.json`) and use it to validate incoming event payloads

**Outcome**: Validation passes for well-formed events; schema violations are caught without writing custom parsing logic for each event type

## Motivation

Non-Python tools (dashboard builders, log analyzers, monitoring integrations) need a machine-readable contract for event payloads. The human-readable `EVENT-SCHEMA.md` doesn't serve that use case.

## Proposed Solution

1. Create `scripts/little_loops/generate_schemas.py` â€” enumerates all 19 event types and outputs one JSON Schema file per type to `docs/reference/schemas/`
2. Commit the generated schema files to `docs/reference/schemas/`
3. Document the regeneration workflow in `CONTRIBUTING.md`

## Design Decision

Schemas live in `docs/reference/schemas/` (documentation artifact, not in the Python package) because:
- Primary consumers are non-Python tools that read from the repo, not from a pip install
- No runtime validation use case has been identified yet
- Avoids package weight and a two-place maintenance problem
- If programmatic access becomes needed, schemas can be moved into `scripts/little_loops/schemas/` at that point via `importlib.resources`

**Wire Format (Critical for Schema Structure)**

`LLEvent.to_dict()` at `events.py:42-48` **flattens** payload into the top-level dict:

```json
{"event": "<type>", "ts": "<ISO8601>", ...payload_fields_inlined}
```

JSON Schemas must validate this **flat** structure â€” `event` and `ts` are top-level required fields on every schema. There is **no `payload` wrapper key** in the wire format. Exception: `evaluate` emits `{"event": "evaluate", "ts": "...", "type": "<evaluator_type>", "verdict": "..."}` â€” `type` here is a payload field at the wire level, not the event identifier.

**JSON Schema Convention** (from `scripts/little_loops/fsm/fsm-loop-schema.json`)

- `"$schema": "http://json-schema.org/draft-07/schema#"`
- `"$id": "little-loops://<filename>.json"` â€” use `"little-loops://event-<event_type>.json"` for event schemas
- Include `title` and `description` fields

**Similar Patterns**
- `scripts/little_loops/cli/docs.py:9-98` â€” closest analogue: `main_verify_docs() -> int` with lazy imports + `argparse.RawDescriptionHelpFormatter`
- `scripts/little_loops/cli/history.py:269-270` â€” `Path.write_text(json.dumps(..., indent=2))` + `mkdir(parents=True, exist_ok=True)` before writing output

## Implementation Steps

1. Read `docs/reference/EVENT-SCHEMA.md` to extract payload field definitions for all 19 event types
2. Create `scripts/little_loops/generate_schemas.py`: build one JSON Schema dict per event type using `json.dumps()`, write to `docs/reference/schemas/<event_type>.json` (replace `.` with `_` in filenames for filesystem safety, e.g. `state_issue_completed.json`)
3. Run the script and commit the 19 generated `.json` files to `docs/reference/schemas/`
4. Add a `CONTRIBUTING.md` note: "When adding a new event type, update `EVENT-SCHEMA.md` and run `python scripts/little_loops/generate_schemas.py` to regenerate schemas"
