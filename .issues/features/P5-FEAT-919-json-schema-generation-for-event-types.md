# FEAT-919: JSON Schema Generation for Event Types

## Summary

Generate JSON Schema files for all 19 `LLEvent` types so non-Python tools can validate event payloads without parsing `docs/reference/EVENT-SCHEMA.md` by hand. Schemas are committed to `docs/reference/schemas/` alongside the human-readable schema doc and regenerated by a script when events change.

## Context

Split from FEAT-916 (Extension SDK). FEAT-916 resolved that the scaffolded `extension.py` skeleton links to `docs/reference/EVENT-SCHEMA.md` in a comment — no schema file reference in templates. This issue covers the schema generation artifact itself, targeting non-Python tooling authors as the primary consumer.

## Current Behavior

No machine-readable schema exists. Extension authors and non-Python tools must parse `docs/reference/EVENT-SCHEMA.md` manually to understand event payload structure.

## Expected Behavior

- `docs/reference/schemas/` contains one `.json` schema file per event type (19 total)
- A generation script at `scripts/little_loops/generate_schemas.py` (or a CLI flag) regenerates all schema files from the known event type catalog
- Schema files are valid JSON Schema (draft-07 or later)

## Motivation

Non-Python tools (dashboard builders, log analyzers, monitoring integrations) need a machine-readable contract for event payloads. The human-readable `EVENT-SCHEMA.md` doesn't serve that use case.

## Proposed Solution

1. Create `scripts/little_loops/generate_schemas.py` — enumerates all 19 event types and outputs one JSON Schema file per type to `docs/reference/schemas/`
2. Commit the generated schema files to `docs/reference/schemas/`
3. Document the regeneration workflow in `CONTRIBUTING.md`

## Design Decision

Schemas live in `docs/reference/schemas/` (documentation artifact, not in the Python package) because:
- Primary consumers are non-Python tools that read from the repo, not from a pip install
- No runtime validation use case has been identified yet
- Avoids package weight and a two-place maintenance problem
- If programmatic access becomes needed, schemas can be moved into `scripts/little_loops/schemas/` at that point via `importlib.resources`

## Integration Map

### Files to Create
- `scripts/little_loops/generate_schemas.py` — generation script
- `docs/reference/schemas/*.json` — 19 schema files (one per event type)

### Key Reference
- `docs/reference/EVENT-SCHEMA.md` — source of truth for all 19 event types across 5 subsystems; schemas must be enumerated from here, **not** from dataclass introspection
- `scripts/little_loops/events.py:28-64` — `LLEvent` is a **flat dataclass** with `type: str` and `payload: dict[str, Any]`; no typed subclasses to introspect. Wire format: `{"event": type, "ts": timestamp, ...payload}`

### Event Type Catalog (19 total, 5 subsystems)
- **FSM Executor (11)**: `loop_start`, `state_enter`, `route`, `action_start`, `action_output`, `action_complete`, `evaluate`, `retry_exhausted`, `handoff_detected`, `handoff_spawned`, `loop_complete`
- **FSM Persistence (1)**: `loop_resume`
- **StateManager (2)**: `state.issue_completed`, `state.issue_failed`
- **Issue Lifecycle (4)**: `issue.failure_captured`, `issue.closed`, `issue.completed`, `issue.deferred`
- **Parallel Orchestrator (1)**: `parallel.worker_completed`

### Schema Generation Notes
- No `jsonschema` generation tooling exists in the codebase — write schemas as hand-constructed dicts using `json.dumps()`
- Each schema should cover the known top-level fields for that event type based on `EVENT-SCHEMA.md`; `payload` fields that vary can be documented as `additionalProperties: true`

### Tests
- `scripts/tests/test_generate_schemas.py` — verify all 19 types produce valid output files; spot-check field names on 2–3 representative types

### Documentation
- `CONTRIBUTING.md` — add note on regenerating schemas when adding a new event type

## Implementation Steps

1. Read `docs/reference/EVENT-SCHEMA.md` to extract payload field definitions for all 19 event types
2. Create `scripts/little_loops/generate_schemas.py`: build one JSON Schema dict per event type using `json.dumps()`, write to `docs/reference/schemas/<event_type>.json` (replace `.` with `_` in filenames for filesystem safety, e.g. `state_issue_completed.json`)
3. Run the script and commit the 19 generated `.json` files to `docs/reference/schemas/`
4. Add a `CONTRIBUTING.md` note: "When adding a new event type, update `EVENT-SCHEMA.md` and run `python scripts/little_loops/generate_schemas.py` to regenerate schemas"

## Acceptance Criteria

- [ ] 19 schema files exist in `docs/reference/schemas/`, one per event type
- [ ] Each schema is valid JSON Schema and covers the known top-level payload fields from `EVENT-SCHEMA.md`
- [ ] Generation script can be re-run idempotently (overwrites existing files)
- [ ] `CONTRIBUTING.md` documents the regeneration workflow
