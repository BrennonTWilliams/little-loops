# FEAT-953: as-a-user eval harness skill from issue IDs

## Summary

Create a new `/ll:create-eval-from-issues` skill that accepts one or more Issue IDs (open or completed), reads each issue's context and acceptance criteria, and generates a ready-to-run FSM eval harness YAML. The harness exercises the described feature **as a real user would** and evaluates the quality of that experience ‚Äî not whether the issue was implemented. This is the bridge between issue-driven development and user-perspective quality evaluation loops.

## Current Behavior

There is no automated way to go from an issue file to a user-perspective eval harness. Developers must hand-author harness YAML, manually translate issue language into user scenarios and evaluation criteria, and reason about what "success as a user" means for each feature.

## Expected Behavior

Running `/ll:create-eval-from-issues FEAT-919 ENH-950` should:
1. Read each named issue file and extract relevant context (title, expected behavior, use case, acceptance criteria)
2. Synthesize a natural-language **execute prompt** describing how a user would exercise the feature (drawn from Expected Behavior / Use Case sections)
3. Synthesize `llm_structured` **evaluation criteria** describing what a successful user experience looks like (drawn from Acceptance Criteria / Use Case outcome)
4. Generate a harness YAML with both prompts embedded and write it to `.loops/eval-harness-<slug>.yaml`
5. Validate the harness via `ll-loop validate` and print the result

For a single issue: produce a Variant A (single-shot) harness with `initial: execute`.
For multiple issues: produce a Variant B (multi-item) harness with a `discover` state iterating over each issue ID.

## Use Case

**Who**: A developer using little-loops on any project (web app, CLI tool, API service) who wants automated user-perspective quality evaluation of their features.

**Workflow A ‚Äî Ongoing quality eval loop**:
A team runs a recurring loop that exercises their app as a user would, evaluates the quality, and feeds results back into issue tracking:
```
Use app as user (e.g. Playwright) ‚Üí Evaluate experience ‚Üí Create/update Issues ‚Üí Refine Issues ‚Üí Implement Issues ‚Üí Repeat
```
The eval harness generated by this skill is the "use app as user ‚Üí evaluate experience" portion of that loop. The developer runs `/ll:create-eval-from-issues` with the relevant feature issue IDs to generate a harness covering those features, then wires it into the larger loop.

**Workflow B ‚Äî Red-green-refactor for evals**:
A developer wants to validate feature quality before and after implementation ‚Äî analogous to red-green-refactor in TDD:
1. Run `/ll:create-eval-from-issues` on open issue IDs ‚Üí generates eval harness
2. Run the harness ‚Üí observe poor/failing eval results (red)
3. Implement the issues (via `/ll:manage-issue` or manually)
4. Re-run the harness ‚Üí observe improved eval results (green)

**Goal**: Generate a valid FSM harness YAML that drives user-perspective evaluation of the features described in the given issues ‚Äî without hand-authoring YAML or manually translating issue language into eval criteria.

**Outcome**: A `.loops/eval-harness-<slug>.yaml` file written to disk, validated by `ll-loop validate`, ready to run immediately with `ll-loop run`.

## Motivation

The `check_skill` evaluation gate is the highest-fidelity phase in the harness pipeline ‚Äî it's the only phase that evaluates from a real user's perspective. But authoring a good `check_skill` harness requires two hard things: (1) articulating what user interaction exercises the feature, and (2) articulating what a successful experience looks like. Both of those already exist in an issue file ‚Äî in the Expected Behavior and Acceptance Criteria sections respectively. This skill closes the translation gap automatically.

This also enables the "issue-to-harness-to-loop" path: capture issues ‚Üí generate eval harness ‚Üí run quality loop unattended.

## Proposed Solution

### Skill: `/ll:create-eval-from-issues`

**Arguments**: one or more issue IDs (e.g., `FEAT-919`, `ENH-950`)
**Output**: `.loops/eval-harness-<slug>.yaml`

**Steps:**

1. **Resolve issue files** ‚Äî use `ll-issues show <ID> --json` to resolve each ID to a file path and basic metadata; support both open and completed issues.

2. **Extract evaluation context** ‚Äî from each issue, extract:
   - `title` ‚Äî the feature being evaluated
   - **Expected Behavior** / **Use Case** (goal + outcome) ‚Äî what does a user do and what do they observe?
   - **Acceptance Criteria** ‚Äî what specific conditions signal a successful experience?

3. **Synthesize harness prompts** ‚Äî produce two synthesized prompts per issue:
   - **Execute prompt**: natural-language description of how to exercise the feature as a user would. Source: Expected Behavior steps + Use Case workflow. Example: *"Open the schema builder, provide a sample input object, and trigger schema generation. Observe the output and any feedback shown."* This is `action_type: prompt` ‚Äî the executing agent (Claude or a tool like Playwright) will carry it out using whatever tools are available in the loop context.
   - **Evaluation criteria**: `llm_structured` prompt asking whether the user experience met the issue's success signals. Source: Acceptance Criteria + Use Case outcome. Structured as: did observable condition 1 occur? condition 2? condition N? Answer YES only if all pass.

4. **Select harness variant**:
   - 1 issue ‚Üí Variant A (single-shot) with `initial: execute`
   - 2+ issues ‚Üí Variant B (multi-item) with `discover` state iterating over issue IDs

5. **Generate YAML** following the structure in `docs/guides/AUTOMATIC_HARNESSING_GUIDE.md`:
   - `execute` state: synthesized user-exercise prompt (`action_type: prompt`)
   - `check_skill` state: synthesized `llm_structured` evaluation criteria
   - No `check_invariants` ‚Äî eval harnesses measure experience quality, not code change size

6. **Validate** via `ll-loop validate` before writing and report result.

### Example output (single issue ‚Äî web app context)

*Given FEAT-919: "Add JSON schema generation to the schema builder"*

```yaml
name: "eval-harness-feat-919"
initial: execute
max_iterations: 5
states:
  execute:
    action: >
      Use the JSON schema generation feature as a real user would.
      Open the schema builder, provide a representative sample input object,
      and trigger schema generation. Observe what happens ‚Äî note any errors,
      delays, or unexpected behavior in the output.
    action_type: prompt
    timeout: 300
    next: check_skill

  check_skill:
    action: >
      Evaluate the JSON schema generation experience from FEAT-919 from a
      real user's perspective.
    action_type: prompt
    timeout: 180
    evaluate:
      type: llm_structured
      prompt: >
        Did the JSON schema generation feature (FEAT-919) deliver a satisfying
        user experience? Assess all of the following:
        (1) The schema builder accepted a sample input without errors
        (2) A valid, readable JSON schema was returned
        (3) The interaction felt smooth ‚Äî no confusing errors, no unexpected delays
        Answer YES only if all three conditions were clearly met.
        Answer NO and specify which condition(s) failed and what was observed.
    on_yes: done
    on_no: execute

  done:
    terminal: true
```

*Note*: The `action_type: prompt` in the `execute` state means the running agent exercises the feature using whatever tools are available in the loop context ‚Äî Playwright MCP, browser tools, CLI invocation, etc. The harness is tool-agnostic; the execute prompt describes user intent, not tool invocation.

### Codebase Research Findings ‚Äî Pass 2 (2026-04-05)

_Added by `/ll:refine-issue` ‚Äî second research pass:_

**CRITICAL: `ll-issues show --json` does NOT return section text**
- `scripts/little_loops/cli/issues/show.py:191-205` ‚Äî `_parse_card_fields()` outputs exactly 13 keys: `{path, title, priority, status, effort, confidence, outcome, summary, integration_files, risk, labels, history, issue_id}`. The `## Expected Behavior`, `## Use Case`, and `## Acceptance Criteria` sections are **not extracted** by this command.
- **Required two-step approach**: call `ll-issues show <ID> --json` to get the `path` field, then read the file at `path` directly to extract Expected Behavior, Use Case, and Acceptance Criteria section text for prompt synthesis.

**`harness-single-shot.yaml` ‚Äî generated harness is purpose-built, not a template copy**
- `scripts/little_loops/loops/harness-single-shot.yaml:93-111` ‚Äî the `check_skill` state is **commented out** in this template; its active path is `execute ‚Üí check_stall ‚Üí check_concrete ‚Üí check_semantic ‚Üí check_invariants ‚Üí done`.
- The eval harness this skill generates is **simpler and purpose-built**: only `execute ‚Üí check_skill ‚Üí done`. The template files are reference for YAML syntax only, not the base to copy from.

**Variant B `discover` state ‚Äî embed IDs at generation time**
- `scripts/little_loops/loops/harness-multi-item.yaml:31-52` ‚Äî the template's `discover` state calls `ll-issues list --json` dynamically. For eval harnesses with known argument IDs, the discover state must embed them at YAML generation time (Python one-liner iterating over the provided ID list), or use sequential chained states (`execute_<id> ‚Üí check_skill_<id>` per issue) to avoid FSM iteration complexity with static lists.

**`check_skill` action type for eval harnesses**
- `docs/guides/AUTOMATIC_HARNESSING_GUIDE.md` ‚Äî `check_skill` requires: `action`, `action_type`, `evaluate.type: llm_structured`, `evaluate.prompt`, `on_yes`, `on_no`; `timeout` strongly recommended.
- Use `action_type: prompt` (not `slash_command`) for synthesized evaluation text. The multi-item template uses `slash_command` for `/ll:ready-issue`; eval harnesses use `prompt` instead.

**FEAT-958 ‚Äî `$ARGUMENTS` required in SKILL.md**
- `P2-FEAT-958` adds `skill_expander.py` to pre-expand skills for `ll-auto`/`ll-parallel` subprocess contexts. The new SKILL.md must use `$ARGUMENTS` for issue ID inputs so `expand_skill()` substitutes them correctly in automation contexts.

### Codebase Research Findings ‚Äî Pass 3 (2026-04-05)

_Added by `/ll:refine-issue` ‚Äî third research pass:_

**Variant B discover state ‚Äî exact FSM structure (resolve ambiguity from Pass 2)**
- `scripts/little_loops/loops/harness-multi-item.yaml:31-52` ‚Äî `discover` uses `fragment: shell_exit` (not `action_type: shell`); this routes on exit code (0 = `on_yes`, non-zero = `on_no`). Also requires `capture: current_item` ‚Äî downstream states reference `${captured.current_item.output}`.

- **Resolved approach for hardcoded IDs**: embed the provided IDs in the `discover` action as a Python inline that tracks processed items via a temp file. Example skeleton (generated at YAML write time with actual IDs substituted):
  ```yaml
  discover:
    action: |
      python3 -c "
      import sys
      ids = ['FEAT-919', 'ENH-950']
      pf = '/tmp/eval-harness-feat-919-enh-950-processed.txt'
      try: processed = open(pf).read().split()
      except FileNotFoundError: processed = []
      remaining = [i for i in ids if i not in processed]
      if not remaining: sys.exit(1)
      print(remaining[0])
      "
    fragment: shell_exit
    capture: current_item
    on_yes: execute
    on_no: done
    on_error: done
  advance:
    action: |
      echo "${captured.current_item.output}" >> /tmp/eval-harness-feat-919-enh-950-processed.txt
    action_type: shell
    next: discover
  ```

**Harness top-level keys ‚Äî `category` field required**
- `scripts/little_loops/loops/harness-single-shot.yaml:14-26` ‚Äî top-level keys in order: `name`, `category`, `description`, `initial`, `max_iterations`, `timeout`, `import`, `states`.
- Set `category: harness` to match built-in templates. Verify `category` is in `KNOWN_TOP_LEVEL_KEYS` at `scripts/little_loops/fsm/validation.py:77` before using a novel value like `eval-harness`.

**`allowed-tools` exact format ‚Äî single list entry with comma-separated patterns**
- `skills/create-loop/SKILL.md:6-8` ‚Äî one list item, all permitted `Bash` command prefixes comma-separated inside parens:
  ```yaml
  allowed-tools:
    - Bash(ll-issues:*, ll-loop:*, mkdir:*)
  ```
  Python invocations (`python3 -c ...`) are wrapped in shell commands; if `python3` must be called directly, add `python3:*` to the pattern list. Do NOT list each prefix as a separate list entry.

**Slug generation ‚Äî inline shell pattern for skill body**
- `scripts/little_loops/issue_parser.py:99-110` ‚Äî canonical `slugify()`: strips non-word chars, collapses hyphens/whitespace to `-`, lowercases. Not directly callable from SKILL.md shell body.
- Inline shell equivalent for harness filename: `echo "${ISSUE_IDS[*]}" | tr '[:upper:]' '[:lower:]' | tr ' ' '-'` ‚Üí e.g., `feat-919-enh-950`. Filename: `eval-harness-feat-919-enh-950.yaml`.

**`$ARGUMENTS` multi-ID array collection ‚Äî no existing precedent; use array-append pattern**
- Existing skills using `for token in $ARGUMENTS` (`confidence-check/SKILL.md:29-76`, `issue-size-review/SKILL.md:37-65`) overwrite a single `ISSUE_ID` variable ‚Äî designed for one ID only.
- For multi-ID collection, extend to array-append:
  ```bash
  ISSUE_IDS=()
  for token in $ARGUMENTS; do
    case "$token" in
      --*) ;; # skip flags
      *) ISSUE_IDS+=("$token") ;;
    esac
  done
  ```
  After parsing: `${#ISSUE_IDS[@]}` = count; `${ISSUE_IDS[@]}` = all IDs; `${ISSUE_IDS[0]}` = first.
- FEAT-958's `_substitute_arguments()` will do `" ".join(args)` ‚Äî so `ll-auto` passing `["FEAT-919", "ENH-950"]` produces `"FEAT-919 ENH-950"` as the `$ARGUMENTS` string, correctly iterated by the loop.
