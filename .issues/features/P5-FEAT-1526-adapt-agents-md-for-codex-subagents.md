# FEAT-1526: Adapt agents/*.md for Codex Subagents

## Summary

Codex CLI's subagent system (`.codex/agents/*.toml`, see
[developers.openai.com/codex/subagents](https://developers.openai.com/codex/subagents))
is the structural analog of Claude Code's `.claude/agents/*.md`. ll currently
ships only the Claude-shaped files, leaving Codex users without our custom
agents (`code-reviewer`, `Explore`, `Plan`, etc.). This issue adds a one-shot
adapter — `ll-adapt-agents-for-codex` — that generates `.codex/agents/<name>.toml`
from each `agents/*.md`, mirroring the established `ll-adapt-skills-for-codex`
(FEAT-1486) pattern.

## Current Behavior

- `agents/*.md` define ll's agents with Claude-shaped frontmatter
  (`name`, `description`, `tools`, optional `model`).
- Under Codex, `CodexRunner.build_streaming(agent=...)` emits a
  `CapabilityNotSupported` warning and drops the parameter. There is no native
  way for a Codex user to invoke ll-defined agents.
- `docs/codex/README.md` lists `--agent` as deferred.
- `docs/reference/HOST_COMPATIBILITY.md` shows `agent_select` for Codex as ✗.

## Expected Behavior

After this issue:
- Each `agents/<name>.md` has a corresponding `.codex/agents/<name>.toml`
  generated by `ll-adapt-agents-for-codex --apply`.
- Each TOML file contains `name`, `description`, `developer_instructions`
  (the agent body), and (when present in source) `model` / `mcp_servers`.
- Codex sessions in repos with `.codex/agents/` present can spawn ll-defined
  agents through Codex's normal in-conversation subagent mechanism.
- `docs/reference/HOST_COMPATIBILITY.md` updates Codex `agent_select` from
  ✗ to ✓ (or `(partial — model-spawned)` to flag the invocation-model
  difference).

## Motivation

This feature would:
- Close the last meaningful Claude/Codex parity gap in ll (skills and
  commands shipped via FEAT-1486/FEAT-1493; agents are the remaining hole).
- Give Codex users access to ll's curated agents (`Explore`, `Plan`,
  `code-reviewer`, ll-namespaced research agents) instead of falling back
  to Codex's three built-ins (`default`, `worker`, `explorer`).
- Follow the established cross-host pattern in this repo: **generate
  per-host native files from a single source-of-truth**, rather than
  attempting to translate at the CLI invocation layer (which is structurally
  impossible — Codex spawns subagents model-internally, not via flag).
- Make `CodexRunner`'s "agent dropped with warning" behavior actionable —
  the warning text already points at this adapter.

## Use Case

**Who**: An ll user whose primary host is Codex CLI, working in a repo that
defines custom agents in `.claude/agents/` or `agents/`.

**Context**: They open a Codex session and ask "use the code-reviewer agent
to review this diff". Today, nothing happens — Codex has never heard of
`code-reviewer`.

**Goal**: Codex's orchestrator recognizes ll-defined agents and spawns them
when contextually appropriate.

**Outcome**: After running `ll-adapt-agents-for-codex --apply` once,
`.codex/agents/code-reviewer.toml` exists and Codex spawns the agent during
relevant tasks.

## Research Notes

- **Established pattern in this repo**: `scripts/little_loops/cli/adapt_skills_for_codex.py`
  is the canonical model — three-file CLI registration, dry-run by default,
  `_find_plugin_root()` for source discovery, additive non-destructive writes.
- **Codex subagent TOML schema**: per
  [developers.openai.com/codex/subagents](https://developers.openai.com/codex/subagents),
  fields are `name`, `description`, `developer_instructions`, optional
  `model`, optional `sandbox_mode`, optional `mcp_servers`. The body of the
  Claude-shaped `agents/*.md` (after frontmatter) maps to `developer_instructions`.
- **Invocation model gap (cannot be closed)**: Claude Code dispatches sub-agents
  at the platform level via the Agent tool (`subagent_type: <name>`); Codex
  spawns subagents model-internally during a conversation. This adapter closes
  the *definition* gap but the *invocation* gap remains — callers passing
  `agent=` to `CodexRunner` still get a warning + drop, because Codex provides
  no mechanism to pre-select a subagent for a one-shot `codex exec`. This is
  consistent with how every surveyed cross-host plugin handles it (see prior-art
  notes below).
- **Prior art surveyed**:
  - [EveryInc/compound-engineering-plugin](https://github.com/EveryInc/compound-engineering-plugin)
    — ships separate per-host agent files via a Bun installer step. Confirms
    "generate native files, don't translate at runtime" is the pragmatic pattern.
  - [mco-org/mco](https://github.com/mco-org/mco) — neutral orchestration layer
    that owns its own agent registry and writes per-host adapters. More
    architecture than ll needs; we already own `agents/*.md` as source-of-truth.
  - `claude-to-codex` and `sync-claude-skills-to-codex` community tools
    explicitly handle *skills only*, not agents.
- **What we don't do**: prompt-injection of the persona's `developer_instructions`
  into the user prompt prefix. Tempting because it would honor `agent=` under
  Codex, but: (a) collides with Codex's native subagent spawning once TOML files
  exist, (b) loses tool-allowlist semantics entirely, (c) diverges agent
  behavior between hosts in invisible ways. Rejected.

## Implementation Steps

1. Audit `agents/*.md` to enumerate sources and identify frontmatter fields
   in use (`name`, `description`, `tools`, `model`).
2. Write `scripts/little_loops/cli/adapt_agents_for_codex.py`:
   - Walk `agents/*.md` via `_find_plugin_root()` + glob.
   - Parse frontmatter with `yaml.safe_load`.
   - Emit TOML via `tomli_w` (stdlib `tomllib` is read-only); add to
     `scripts/pyproject.toml` dependencies if not already present.
   - Write to `<repo>/.codex/agents/<name>.toml` with `# generated by ll-adapt-agents-for-codex`
     header for idempotency detection.
   - Dry-run mode shows diff; `--apply` writes.
3. Register CLI (three-file pattern: module + `__init__.py` re-export + pyproject entry).
4. Run adapter against the real `agents/` dir and commit the generated
   `.codex/agents/*.toml` files.
5. Update docs (HOST_COMPATIBILITY, codex/README, codex/usage, CLI.md, help.md,
   CLAUDE.md, areas.md, README counts).
6. Write tests modeled on `test_adapt_skills_for_codex.py`.
7. Manual smoke test: in a Codex session, ask the model to use one of the
   ll-defined agents and verify it spawns.

## API/Interface

```bash
# Default: dry-run, prints generated TOML to stdout
ll-adapt-agents-for-codex

# Write .codex/agents/<name>.toml
ll-adapt-agents-for-codex --apply

# Restrict to a single agent
ll-adapt-agents-for-codex --apply --only code-reviewer
```

Python: `main_adapt_agents_for_codex() -> int` returning a process exit code.

## Impact

- **Priority**: P5 — Codex host is exploratory; Claude remains primary. No
  user is blocked but the gap is the last meaningful one in our cross-host
  story.
- **Effort**: Small-to-medium — the structural template exists
  (`adapt_skills_for_codex.py`); main novelty is TOML emission and the
  `.codex/agents/` target directory.
- **Risk**: Low — additive; generated files live in `.codex/agents/`,
  independent of `agents/*.md` sources; idempotent; no existing invocation
  path changes; Codex users opt in by running the CLI.
- **Breaking Change**: No.
