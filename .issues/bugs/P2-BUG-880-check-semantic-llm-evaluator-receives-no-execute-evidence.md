# BUG-880: Wizard-generated `check_semantic` provides no evidence to LLM evaluator

## Summary

The `check_semantic` state generated by the harness wizard uses `action: echo '...'` (a shell echo) as its action. The `llm_structured` evaluator receives only the echo text as `action_output` ‚Äî not the output from the `execute` state that ran the skill.

In `evaluators.py:evaluate_llm_structured`, the `output` parameter is constructed as:

```python
user_prompt = f"{effective_prompt}\n\n<action_output>\n{truncated}\n</action_output>"
```

For `check_semantic`, `truncated` = `"Evaluating refinement quality\n"` (the echo). The LLM evaluator has no access to what `execute` actually produced.

The `EvaluateConfig` schema supports a `source` field (`executor.py:876-882`) that allows redirecting the eval input via interpolation (e.g. `source: "${prev.output}"`), but the wizard does not generate this field. Without it, `check_semantic` evaluates blind ‚Äî it can only answer the YES/NO question based on priors, not actual evidence.

## Current Behavior

The wizard generates `check_semantic` YAML without a `source:` field in its `evaluate` block. When `check_semantic` runs, `executor._evaluate()` uses `eval_input = raw_output` (the current action's echo output) as the LLM context. The evaluator receives `"Evaluating refinement quality\n"` as evidence ‚Äî not the actual output from `execute`.

## Expected Behavior

The wizard should generate `check_semantic` with:
- `capture: execute_result` on the `execute` state
- `source: "${captured.execute_result.output}"` in `check_semantic`'s `evaluate` block

This ensures the `llm_structured` evaluator receives actual skill output as evidence for its YES/NO quality judgment.

## Root Cause

`skills/create-loop/` wizard generates `check_semantic` YAML without a `source: "${prev.output}"` field. The `executor._evaluate()` method defaults to `eval_input = raw_output` (the current action's output) when `source` is absent.

### Codebase Research Findings

_Added by `/ll:refine-issue` ‚Äî based on codebase analysis:_

**Critical correction: `${prev.output}` is the wrong interpolation expression.**

`self.prev_result` is set at `executor.py:666-670` from each state's completed `action_result` and is **overwritten on every state transition**. In the harness chain `execute ‚Üí check_stall ‚Üí check_concrete ‚Üí check_semantic`, by the time `check_semantic` runs, `${prev.output}` resolves to `check_concrete`'s output ‚Äî not `execute`'s.

All working examples in the codebase use `capture:` + `${captured.*}` for cross-state output forwarding (e.g., `loops/issue-staleness-review.yaml:36-47`, `loops/sprint-build-and-validate.yaml:57-64`).

**Correct fix requires two changes per state chain:**
1. Add `capture: execute_result` to the `execute` state
2. Add `source: "${captured.execute_result.output}"` to `check_semantic`'s `evaluate` block

## Steps to Reproduce

1. Run `/ll:create-loop` and select prompt/execute harness type with a `check_semantic` quality gate
2. Inspect the generated YAML ‚Äî observe `check_semantic`'s `evaluate:` block has no `source:` field
3. Also inspect `loops/harness-single-shot.yaml` and `loops/harness-multi-item.yaml` ‚Äî same deficiency
4. Run the harness loop with any skill that produces output
5. Observe `check_semantic` evaluates based only on the echo string, not the skill's actual output

## Impact

- **Priority**: P2 ‚Äî silent quality gate failure; users relying on `check_semantic` receive false confidence that the evaluator verified skill quality
- **Effort**: Small ‚Äî YAML template changes in `skills/create-loop/loop-types.md` and 2 harness YAML files; `source:` field already implemented in `executor.py`
- **Risk**: Low ‚Äî additive YAML changes only; executor already handles the `source` redirect path; existing `TestEvaluateSource` tests cover this behavior
- **Breaking Change**: No ‚Äî `check_semantic` loops currently pass regardless; adding `source:` makes them evaluate correctly
