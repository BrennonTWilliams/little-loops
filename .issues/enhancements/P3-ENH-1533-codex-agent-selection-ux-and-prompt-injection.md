# ENH-1533: Improve Codex agent selection — surface UX warning and implement prompt injection

## Summary

ENH-1531 confirmed Codex has no native `--agent` CLI flag and the gap is permanent until openai/codex#10067 ships. Two improvements are viable regardless:

- **Option A**: Make `CapabilityNotSupported` user-visible — the current Python `warnings.warn` is silent in most invocation paths; Codex users never know their agent hint was dropped.
- **Option B**: Implement prompt injection — read `developer_instructions` from `.codex/agents/<name>.toml` and prepend it to the prompt payload, upgrading `agent_select` from `"unsupported"` to `"partial"` in `describe_capabilities()`.

Both options can land independently; Option B subsumes Option A (no warning fires if the workaround succeeds).

## Current Behavior

`CodexRunner.build_streaming()` emits `CapabilityNotSupported` via `warnings.warn()` and drops the `agent` parameter silently. The warning is a Python-level warning — it only surfaces if the calling code has a `warnings.simplefilter` configured. In `ll-auto`, `ll-parallel`, and `ll-loop`, the warning goes nowhere. The session proceeds with no persona and the user receives no explanation.

## Expected Behavior

**Option A only**: When `agent` is requested and Codex is the host, print a clear stderr message alongside (or instead of) the Python warning:

```
[ll] Warning: Codex does not support --agent at invocation time (ENH-1531).
     Persona hint 'code-reviewer' was dropped. For interactive sessions,
     run `ll-adapt-agents-for-codex --apply` and use `--agent code-reviewer`
     in the Codex TUI.
```

**Option B**: `CodexRunner.build_streaming()` reads `.codex/agents/<name>.toml`, extracts `developer_instructions`, and prepends it to the prompt. No warning fires. `describe_capabilities()` reports `agent_select: partial`. If the `.toml` file does not exist, fall back to Option A behavior (warn + drop).

## Motivation

`ll-auto --agent code-reviewer` (or FSM states with `agent: code-reviewer`) is a first-class workflow on Claude Code. Codex users running the same loops get silent persona loss with no indication. The silent drop is worse UX than any other capability gap in ll — other unsupported flags at least surface `CapabilityNotSupported` somewhere visible.

Option B is the established ll pattern for "host doesn't support X natively — mediate through a file." The `developer_instructions` content is already on disk in `.codex/agents/<name>.toml` (generated by `ll-adapt-agents-for-codex`). The infrastructure is there; it just needs to be wired.

## Proposed Solution

### Option A: stderr notice

In `CodexRunner.build_streaming()`, replace (or supplement) the `warnings.warn` call with a `print(..., file=sys.stderr)` that names the dropped agent and points to the interactive workaround.

### Option B: Prompt injection

> **Selected:** Option B (Prompt injection) — subsumes Option A as the TOML-absent fallback, actually preserves persona behavior (not just signals loss), and reuses the established ENH-1530 file-mediation pattern with zero new dependencies.

```python
# CodexRunner.build_streaming(), when agent is not None
toml_path = Path(".codex/agents") / f"{agent}.toml"
if toml_path.exists():
    data = tomllib.loads(toml_path.read_text())
    instructions = data.get("developer_instructions", "").strip()
    if instructions:
        prompt = f"[Persona: {agent}]\n{instructions}\n\n---\n\n{prompt}"
        # no warning — workaround succeeded
    else:
        _emit_agent_warning(agent)
else:
    _emit_agent_warning(agent)
```

`_emit_agent_warning` handles the Option A stderr output for the fallback path.

Update `describe_capabilities()`:
- Change `CapabilityEntry("agent_select", status="unsupported", ...)` to `status="partial"` with note explaining prompt injection
- `HostCapabilities.agent_select` bool stays `False` (partial ≠ full CLI parity)

### Decision Rationale

Decided by `/ll:decide-issue` on 2026-05-16.

**Selected**: Option B (Prompt injection, with Option A stderr notice as the TOML-absent fallback)

**Reasoning**: Option B subsumes Option A — the fallback path inside `_inject_agent_persona()` calls `_emit_agent_warning()` whenever the `.codex/agents/<name>.toml` lookup misses, so choosing B delivers A's stderr warning behavior for free while also implementing the actual fix. The issue's stated motivation is silent persona loss, and Option A alone only signals that loss without preventing it; only B preserves the persona behavior Codex users expect from `--agent` workflows. All infrastructure is already in place: 8 `.codex/agents/*.toml` files exist with `developer_instructions` at line 6 (generated by `scripts/little_loops/cli/adapt_agents_for_codex.py:87`), `tomllib` is Python 3.11+ stdlib (`pyproject.toml:11`), and ENH-1530's `build_blocking_json` temp-file bridge at `host_runner.py:467-484` is a direct shape precedent for both the file-mediation implementation and the `"unsupported" → "partial"` upgrade in `describe_capabilities()` (line 530-535).

Friction: `host_runner.py` currently has zero `print` statements, so A introduces the first imperative I/O side-effect into a module designed around pure builder methods. Standalone value is limited — it informs the user but does not solve persona loss.
- **Option B**: Direct precedent in `CodexRunner.build_blocking_json()` (ENH-1530 temp-file bridge at `host_runner.py:467-484`) and matching `describe_capabilities()` `"partial"` upgrade pattern at `host_runner.py:530-535`. Generator at `adapt_agents_for_codex.py:87` already writes the consumed key. Pattern C no-warning test idiom already exists at `test_host_runner.py:277-286`. Only genuine novelty: first `tomllib` import in `scripts/` (stdlib, no dependency) and first prompt-mutation inside a `build_*` method. Single production call site (`subprocess_utils.py:263` via `run_claude_command`), so injection is transparent to all callers. Working-dir gap flagged in the issue's research findings is resolved by `(working_dir or Path.cwd()) / ".codex/agents" / f"{agent}.toml"` — mirrors existing `git_path` working-dir pattern at `host_runner.py:435-442`.

**Tiebreaker note**: Although Option A scores 1 point higher on raw rubric, the rubric measures implementation cost in isolation. Because B's fallback path includes A's full behavior, the realistic comparison is "A alone" vs "A + persona injection". B wins on delivered value at modest additional cost (~2 hr vs ~30 min) with no new dependencies and direct precedent.

### Codebase Research Findings

_Added by `/ll:refine-issue` — verified against actual code (2026-05-16):_

- **TOML structure confirmed**: `.codex/agents/<name>.toml` files (8 present, e.g. `codebase-analyzer.toml:6`) use `developer_instructions = """ ... """` — a multi-line triple-quoted string. `tomllib.loads(path.read_text())` returns it as a plain `str`, so `.get("developer_instructions", "").strip()` works as written in the pseudocode.
- **tomllib availability**: `pyproject.toml:11` pins `requires-python = ">=3.11"`, so `tomllib` is stdlib — no new dependency needed.
- **Generator already produces the key**: `scripts/little_loops/cli/adapt_agents_for_codex.py:87` writes the `developer_instructions` field for every adapted agent; Option B has zero dependency on changes to the generator.
- **Existing warning sites verified**: `CodexRunner.build_streaming()` emits `CapabilityNotSupported` for `agent` at `host_runner.py:403-411` and for `tools` at `host_runner.py:412-419`. `describe_capabilities()` declares `agent_select="unsupported"` at `host_runner.py:517-522`. All anchor references in this issue match the current source.
- **Working-dir path resolution (NEW gap surfaced)**: `build_streaming()` accepts `working_dir: Path | None`. The Option B pseudocode uses `Path(".codex/agents") / f"{agent}.toml"`, which resolves against the caller's CWD — not `working_dir`. In `ll-parallel` worktrees the worktree IS the CWD when the runner is invoked, so plain relative-path resolution works there. But for callers that pass `working_dir` explicitly while running from a different CWD (e.g. a custom orchestrator), the lookup will miss the worktree's `.codex/agents/`. **Recommendation**: resolve as `(working_dir or Path.cwd()) / ".codex/agents" / f"{agent}.toml"` so persona injection follows the same working-dir contract as the rest of `build_streaming()`. Mirror this in tests by exercising both `working_dir=None` and `working_dir=<tmp_path>` cases.
